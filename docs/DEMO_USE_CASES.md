# Αναλυτικά use cases demo — Horeca Source (MVP)

Αυτό το έγγραφο είναι για **έλεγχο ποιότητας στο demo** και για **παρουσίαση** σε χρήστες ή stakeholders. Κάθε σενάριο είναι γραμμένο σε γλώσσα **αρχάριου**: τι κάνεις, γιατί, τι πρέπει να δεις.

**Σύνδεση με υπάρχον υλικό:** τα σύντομα checklist βρίσκονται στο [`TESTING_GUIDE.md`](TESTING_GUIDE.md). Εδώ είναι η **βαθιά ανάλυση** με σκοπό και αναμενόμενα.

---

## 1. Πώς να χρησιμοποιήσεις αυτό το αρχείο

### 1.1 Τι περιλαμβάνει κάθε use case

| Πεδίο | Τι σημαίνει |
|--------|-------------|
| **Κωδικός** | Γρήγορη αναφορά (π.χ. «UC-B02») |
| **Σκοπός** | Γιατί το ελέγχουμε· τι «αποδεικνύει» στην παρουσίαση |
| **Προϋποθέσεις** | Τι πρέπει να ισχύει πριν ξεκινήσεις (λογαριασμός, πλάνο, server) |
| **Βήματα** | Αριθμημένη ροή · ακολούθησέ τα με τη σειρά |
| **Αναμενόμενο** | Τι είναι **επιτυχία**· αν κάτι διαφέρει, καταγράφεις bug |
| **Ιδέα για την παρουσίαση** | Μία πρόταση τι να πεις δυνατά |

### 1.2 Σύμβολα γρήγορης αναφοράς

| Όριο / έννοια | Τι είναι στο demo |
|----------------|-------------------|
| **Δωρεάν πλάνο** | Μέχρι **10 παραγγελίες / μήνα**, **3 αποθηκευμένοι προμηθευτές**, ιστορικό **τελευταίες 30 ημέρες** (σε σχετικές λίστες), **1 κατάστημα**, **1 θέση ομάδας** ανά ταυτότητα των features στο app |
| **Pro πλάνο (mock)** | Αναβάθμιση από την οθόνη συνδρομής **χωρίς πραγματική χρέωση** — dev endpoint |
| **Paywall** | Μήνυμα ότι λειτουργία θέλει Pro, με συνήθως σύνδεση στην οθόνη πλάνων |

### 1.3 Demo λογαριασμοί (επαναλαμβάνονται σε κάθε ενότητα)

| Ρόλος | Email | Κωδικός |
|--------|--------|---------|
| Buyer | `buyer@horeca.demo` | `demo1234` |
| Supplier | `supplier@horeca.demo` | `demo1234` |

## 2. Προετοιμασία περιβάλλοντος (πριν από οποιοδήποτε use case)

**Σκοπός:** Να μην «κολλάς» σε παλιό build ή λάθος δίκτυο.

**Βήματα (developer / παρουσιαστής):**

1. Στο Mac, στον φάκελο project: `pnpm dev` — αφήνεις να τρέχει (API + Metro).
2. **iPhone + Xcode:** ίδιο Wi‑Fi με το Mac.
3. Αν χρησιμοποιείς φυσική συσκευή με LAN API: στο `ios/.xcode.env.local` βάλε `EXPO_PUBLIC_API_BASE_URL=http://<IP-Mac>:<πόρτα πλατφόρμας>` όπως στο [`TESTING_GUIDE.md`](TESTING_GUIDE.md).
4. Προαιρετικά πριν σημαντική παρουσίαση: **Xcode → Product → Clean Build Folder** → **Run**.

**Αναμενόμενο:** Η εφαρμογή ανοίγει· το login φτάνει στο API· δεν βλέπεις συνεχόμενα «σφάλμα δικτύου».

# Ομάδα A — Ταυτότητα, είσοδος/έξοδος

### UC-A01 — Πρώτη εντύπωση από την οθόνη καλωσορίσματος

**Σκοπός:** Ο θεατής να καταλάβει αμέσως **δύο κοινές** (κατάστημα vs προμηθευτής).  
**Προϋποθέσεις:** Φρέσκο launch ή όχι συνδεδεμένος χρήστης (ανά ροή της εφαρμογής σου).

**Βήματα**

1. Άνοιξε την εφαρμογή μέχρι να δεις την οθόνη εισόδου / welcome αν εμφανίζεται στη ροή σου.

**Αναμενόμενο**

1. Φαίνεται προσανατολισμός αξίας για **καταστήματα** και για **προμηθευτές** (dual value).
2. Υπάρχουν οι ενέργειες που οδηγούν σε **Σύνδεση** / **Εγγραφή** ανάλογα με το τι δείχνει η τρέχουσα build ροή.

**Ιδέα για παρουσίαση:** «Μία εφαρμογή, δύο προσωποποιήσεις—ο καταστηματάρχης ψωνίζει, ο προμηθευτής εκτελεί.»

---

### UC-A02 — Επιτυχής είσοδος ως Buyer

**Σκοπός:** Ο βασικός ροή «ξεκλείδωμα» του προϊόντος για καταστήματα.

**Προϋποθέσεις:** Τρέχει το API.

**Βήματα**

1. Από την οθόνη εισόδου επίλεξε buyer login (αν χωρίζει ροή)· εισήγαγε `buyer@horeca.demo` / `demo1234`.
2. Επιβεβαίωσε (κουμπί σύνδεσης).

**Αναμενόμενο**

1. Μπαίνεις στο κύριο shell buyer με **5 tabs**: Αρχική, Προμηθευτές, Παραγγελίες, Αγαπημένα, Λογαριασμός.
2. Καμία ατέλειωτη καθυστέρηση· σε σφάλμα εμφανίζεται μήνυμα (όχι άδειο κενό).

**Ιδέα για παρουσίαση:** «Σε 5 δευτερόλεπτα είσαι μέσα στο κατάστημα demo.»

---

### UC-A03 — Λάθος κωδικός (Buyer)

**Σκοπός:** Να φαίνεται **ασφαλής αποτυχία** χωρίς να «σπάει» το UI.

**Προϋποθέσεις:** Στην οθόνη login.

**Βήματα**

1. Email: `buyer@horeca.demo`.
2. Κωδικός: σκόπιμα λάθος (π.χ. `wrong-pass`).
3. Σύνδεση.

**Αναμενόμενο**

1. **Δεν** μπαίνεις στην εφαρμογή.
2. Εμφανίζεται κατανοητό σφάλμα (toast/alert ή κείμενο στην οθόνη).

**Ιδέα για παρουσίαση:** «Δεν διαρρέουμε λεπτομέρειες για το τι ακριβώς ήταν λάθος-αλλά προστατεύουμε τον λογαριασμό.»

---

### UC-A04 — Επιτυχής είσοδος ως Supplier

**Σκοπός:** Ο δεύτερος κεντρικός persona.

**Βήματα**

1. Αν είσαι ήδη συνδεδεμένος: Λογαριασμός → Έξοδος (UC-A05 πρώτα).
2. Σύνδεση με `supplier@horeca.demo` / `demo1234`.

**Αναμενόμενο**

1. **4 tabs**: Πίνακας, Παραγγελίες, Κατάλογος, Λογαριασμός.

**Ιδέα για παρουσίαση:** «Ο προμηθευτής βλέπει διαφορετικό workspace—ίδια τεχνική βάση, άλλα εργαλεία.»

---

### UC-A05 — Αποσύνδεση

**Σκοπός:** Καθαρή μετάβαση προς άλλο demo λογαριασμό ή τέλος συνεδρίας.

**Προϋποθέσεις:** Συνδεδεμένος οποιοσδήποτε ρόλος.

**Βήματα**

1. Tab **Λογαριασμός**.
2. **Έξοδος** (ή ισοδύναμο copy).
3. Επιβεβαίωσε αν ζητηθεί.

**Αναμενόμενο**

1. Επιστροφή σε κατάσταση μη συνδεδεμένου / welcome / login.
2. Δεν μένεις «μισά» μέσα σε δεδομένα χωρίς session.

**Ιδέα για παρουσίαση:** «Ένα tap να αλλάξεις persona για το demo.»

---

### UC-A06 — Νέα εγγραφή (προαιρετικό—δημιουργεί πραγματικό λογαριασμό στη DB σου)

**Σκοπός:** Ολοκληρωμένο funnel «νέος πελάτης».

**Προϋποθέσεις:** Τρέχον API και DB· **άλλο email** από τα demo αν θέλεις να επαναλαμβάνεις.

**Βήματα**

1. Από welcome / login βρες την εγγραφή.
2. Συμπλήρωσε **Επωνυμία**, **business email**, **κωδικό ≥ 8 χαρακτήρες**, επίλεξε ρόλο buyer ή supplier.
3. Ολοκλήρωση.

**Αναμενόμενο**

1. Μετά την επιτυχία εισέρχεσαι στο κατάλληλο tab layout.
2. Το νέο account ξεκινά ως **Δωρεάν** πλάνο (συνήθως auto-enrolment free).

**Ιδέα για παρουσίαση:** «Από μηδέν σε δουλεύοντα λογαριασμό χωρίς κάρτα στο demo.»

**Σημείωση:** Σε παρουσίαση με σταθερή DB, καλύτερα να **μην** δημιουργείς random emails πολλά—χρησιμοποίησε ένα test email που ξέρεις ότι μπορείς να σβήσεις με seed.

---

### UC-A07 — Εγγραφή με ήδη υπάρχον email

**Σκοπός:** Έλεγχος duplicate handling.

**Βήματα**

1. Προσπάθησε να εγγραφείς με `buyer@horeca.demo` (ή υπαρκτό ήδη email).

**Αναμενόμενο**

1. Μήνυμα ότι **υπάρχει ήδη** λογαριασμός· **όχι** silent failure.

---

### UC-A08 — API μη διαθέσιμο (λειτουργικό σφάλμα)

**Σκοπός:** Ο παρουσιαστής ξέρει τι λέει αν «πέσει» ο server μέσα στο meeting.

**Βήματα**

1. Σταμάτα το `pnpm dev` (ή την πλατφόρμα) προσωρινά **ή** δώσε λάθος IP στο env.
2. Προσπάθησε login.

**Αναμενόμενο**

1. Κατάλληλο δικτυακό / μη διαθέσιμο μήνυμα (όχι crash).

**Στην παρουσίαση:** Αν συμβεί, πες ήρεμα: «Στο prod το backend θα είναι hosted—στο LAN demo εξαρτάται από το Mac.»

# Ομάδα B — Buyer: αναζήτηση, αγαπημένα, καλάθι, παραγγελίες

### UC-B01 — Αρχική: γρήγορες ενέργειες και KPI

**Σκοπός:** Να φαίνεται «κέντρο ελέγχου».

**Προϋποθέσεις:** Συνδεδεμένος buyer.

**Βήματα**

1. Tab **Αρχική**.
2. Διάβασε welcome / ρόλο.
3. Δες τις γρήγορες ενέργειες και τα KPI που εμφανίζονται.

**Αναμενόμενο**

1. Υπάρχει διαδρομή προς **Έξοδα** (ή ισοδύναμο).
2. Πρόσφατες παραγγελίες **αν υπάρχουν** δεδομένα· αν είσαι δωρεάν, το χρονικό παράθυρο διάβασης **30 ημερών** σε σχετικές προβολές (ότι δείχνει η εφαρμογή)—συμπίπτει με τις διαγνωστικές σου λίστες στο tab Παραγγελίες.

**Ιδέα για παρουσίαση:** «Από την αρχική καταλαβαίνεις τι «καίει» σήμερα.»

---

### UC-B02 — Προμηθευτές: φίλτρο κατηγορίας

**Σκοπός:** Ανακάλυψη καταλόγου.

**Βήματα**

1. Tab **Προμηθευτές**.
2. Πάτα chip κατηγορίας (π.χ. **Καφές**).
3. Παρατήρησε τη λίστα.

**Αναμενόμενο**

1. Η λίστα ενημερώνεται ανά επιλογή· δεν κολλάει με κενό χωρίς εξήγηση.

---

### UC-B03 — Προφίλ προμηθευτή και αγαπημένο (1ο–3ο)

**Σκοπός:** Social proof + bookmarking.

**Βήματα**

1. Άνοιξε προφίλ προμηθευτή.
2. Πάτα καρδιά / αγαπημένο για **έως 3** διαφορετικούς (δωρεάν).

**Αναμενόμενο**

1. Κάθε επιτυχία ενημερώνει UI· στο tab **Αγαπημένα** εμφανίζονται.
2. Badge τύπου **`n / 3`** στο δωρεάν πλάνο.

---

### UC-B04 — Τέταρτος αγαπημένος στο δωρεάν πλάνο (paywall)

**Σκοπός:** Δείχνεις **εμπορική λογική** χωρίς να «σπας» το demo.

**Προϋποθέσεις:** Buyer **Δωρεάν**· ήδη 3 αγαπημένοι.

**Βήματα**

1. Άνοιξε 4ο προμηθευτή.
2. Προσπάθησε αγαπημένο.

**Αναμενόμενο**

1. Paywall ή αποτυχία με μήνυμα ότι χρειάζεται **Pro** / αναβάθμιση—not silent ignore.

---

### UC-B05 — Κατάλογος προϊόντος και προσθήκη στο καλάθι

**Σκοπός:** Κεντρική αγορά.

**Βήματα**

1. Μπες σε κατάλογο από προφίλ (ή διαδρομή που ήδη χρησιμοποιείς).
2. Άνοιξε προϊόν.
3. Αύξηση ποσότητας / προσθήκη.
4. Άνοιξε το καλάθι.

**Αναμενόμενο**

1. Γραμμές και ποσά ενημερώνονται.
2. Μετά από login τα δεδομένα **σπρώχνονται προς τον server** (sync)—δεν μένουν μόνο τοπικά μετά επαναφόρτωση αν το σενάριο υποστηρίζει persist.

---

### UC-B06 — Πολλαπλοί προμηθευτές στο καλάθι → διαχωρισμένες παραγγελίες

**Σκοπός:** Realistic wholesale—διαφορετικοί επαφές.

**Βήματα**

1. Βάλε προϊόν από **προμηθευτή Α** και από **προμηθευτή Β**.
2. **Checkout**.
3. Ολοκλήρωσε.

**Αναμενόμενο**

1. Δημιουργούνται **χωριστές παραγγελίες** ανά λογική του backend (μιλάμε συνήθως 2 νέες εγγραφές στην λίστα παραγγελιών).

**Ιδέα για παρουσίαση:** «Η εφαρμογή τα χωρίζει αυτόματα—δεν παραγγέλνεις «ένα καλάθι» που δεν υπάρχει στη πραγματικότητα.»

---

### UC-B07 — Μηνιαίο όριο παραγγελιών (Δωρεάν)

**Σκοπός:** freemium pressure—για να κάνουν upgrade στην παραγωγή.

**Προϋποθέσεις:** Λογαριασμός ή ροή που μπορεί να φτάσει **10 ολοκληρωμένες** παραγγελίες στον **τρέχοντα ημερολογιακό μήνα**.

**Πρακτικά στο demo:**

- Το ταχύτερο path είναι ή να έχεις seed data κοντά στο όριο είτε να κάνεις πολλές μικρές παραγγελίες **ή** να εξηγήσεις θεωρητικά στην παρουσίαση αν δεν υπάρχει χρόνος.

**Βήματα**

1. Μετράς παραγγελίες μήνα (badge / λίστα).
2. Προχώρα νέο checkout μέχρι να αγγίξεις το όριο.
3. Προσπάθησε την **επόμενη**.

**Αναμενόμενο**

1. Απόρριψη με **κατανοητό μήνυμα** που δείχνει upgrade (στυλιστικά 402 / paywall ανά υλοποίηση).

---

### UC-B08 — Ιστορικό παραγγελιών και παράθυρο 30 ημερών (Δωρεάν)

**Σκοπός:** Να δείξεις ότι το ιστορικό **δεν είναι άπειρο** στο free.

**Βήματα**

1. Tab **Παραγγελίες**.
2. Αν υπάρχουν παραγγελίες **παλαιότερες** του free παραθύρου, δες αν εμφανίζεται **γραμμή paywall** για πλήρες ιστορικό.

**Αναμενόμενο**

1. Πρόσφατες παραγγελίες εντός παραθύρου φαίνονται κανονικά.
2. Για το «βαθύ» ιστορικό—η εφαρμογή σε σταματάει με πρόταση **Pro**.

# Ομάδα C — Συνδρομή, εξαγωγές, gated actions

### UC-C01 — Περιήγηση πλάνων Δωρεάν vs Pro

**Σκοπός:** Οικονομική διαφάνεια.

**Προϋποθέσεις:** Συνδεδεμένος buyer.

**Βήματα**

1. Tab **Λογαριασμός** → από την κάρτα συνδρομής άνοιξε τα **Πλάνα** / διαχείριση.

**Αναμενόμενο**

1. Ξεχωριστά χαρακτηριστικά ανά tier (σε λίστα bullets όπως στο app).
2. Κατανοητά κουμπιά για **Αναβάθμιση** (mock).

---

### UC-C02 — Αναβάθμιση σε Pro (mock) και αντικατοπτρισμός στο προφίλ

**Σκοπός:** Core monetization narrative στο demo.

**Βήματα**

1. Από τα πλάνα διάλεξε **μηνιαίο ή ετήσιο Pro** ανάλογα με το τι δείχνει το κουμπί.
2. Μετά την επιβεβαίωση, γύρνα σε **Λογαριασμός**.

**Αναμενόμενο**

1. Badge **Pro**.
2. Ξεκλειδώνονται **στο UI** τα σχετικά features (μεταγενέστερα UC επαληθεύουν ροές).

---

### UC-C03 — Εξαγωγή CSV ιστορικού παραγγελιών (Pro)

**Σκοπός:** Operational win για λογιστικά / reporting.

**Προϋποθέσεις:** Buyer **Pro**· υπάρχουν ήδη παραγγελίες (ή όχι αλλά τότε να εξηγείς κενό αρχείο).

**Βήματα**

1. Tab **Παραγγελίες**.
2. **Εξαγωγή ιστορικού** (το κουμπί **δεν** πρέπει να έχει λουκέτο Pro όταν το plan είναι Pro).
3. Αποδοχή share sheet στο iOS.

**Αναμενόμενο**

1. Ανοίγει **native share sheet** με **αρχείο CSV** (π.χ. αποθήκευση σε Αρχεία).
2. **Όχι** alert τύπου «Σύντομα».

---

### UC-C04 — Εξαγωγή δεσμευμένη στο Δωρεάν

**Σκοπός:** Paywall στην πράξη.

**Προϋποθέσεις:** Επίστρεψη σε Δωρεάν (UC-C05) ή διαφορετικό λογαριασμό.

**Βήματα**

1. **Παραγγελίες** → **Εξαγωγή**.

**Αναμενόμενο**

1. Λουκέτο ή paywall που οδηγεί σε συνδρομή—not silent.

---

### UC-C05 — Επιστροφή σε Δωρεάν (κουμπί demo)

**Σκοπός:** Επανάληψη κύκλων χωρίς νέο account.

**Προϋποθέσεις:** Τρέχεις σε **development** ρύθμιση με ενεργό demo endpoint ακύρωσης (όπως στη [`TESTING_GUIDE.md`](TESTING_GUIDE.md)).

**Βήματα**

1. Λογαριασμός → Συνδρομή → **Demo: επιστροφή σε Δωρεάν άμεσα**.
2. Επιβεβαίωση σε alert αν εμφανιστεί.

**Αναμενόμενο**

1. Το προφίλ δείχνει ξανά **Δωρεάν**.
2. Ξανα εμφανίζεται κλείδωμα στα gated features που εξαρτιούνται από Pro.

---

### UC-C06 — Εξαγωγή Εξόδων (CSV) αν το UI το επιτρέπει για το τρέχον πλάνο

**Σκοπός:** Στα **Έξοδα** η εξαγωγή ακολουθεί **ίδια φιλοσοφία gating**.

**Προϋποθέσεις:** Buyer· οθόνη **Έξοδα** (`spending`).

**Βήματα**

1. Άνοιξε τα Έξοδα από **Αρχική** ή από **Λογαριασμός**.
2. Αν είναι ενεργό το στοιχείο εξαγωγής για το πλάνό σου, πάτησέ το.

**Αναμενόμενο**

1. **Pro + ξεκλείδωτο:** share sheet με CSV ή ισοδύναμο.
2. **Δωρεάν ή κλειδωμένο:** εμφανίζεται paywall / μήνυμα upgrade—not κενός κώδικας.

# Ομάδα D — Τοποθεσίες και ομάδα

### UC-D01 — Κατάστημα & ομάδα από Λογαριασμός

**Σκοπός:** Multi-site enterprise story.

**Βήματα**

1. Λογαριασμός → **Καταστήματα & ομάδα**.
2. Δες τη λίστα· πάτα **Πίσω** πάνω πάνω.

**Αναμενόμενο**

1. Μπαίνεις στη λίστα καταστημάτων / προσκλήσεων όπως ορίζει το seed.
2. Επιστροφή με **Πίσω** χωρίς να κολλάς.

---

### UC-D02 — Ενεργό κατάστημα (chips στην Αρχική / Παραγγελίες)

**Σκοπός:** Φιλτράρισμα παραγγελιών ανά εγκατάσταση όταν υπάρχουν πολλές τοποθεσίες.

**Προϋποθέσεις:** Λογαριασμός με δυνατότητα πολλών τοποθεσιών ή seed που το επιδεικνύει.

**Βήματα**

1. Επίλεξε «Ενεργό κατάστημα» chip σε **Αρχική** ή **Παραγγελίες**.
2. Παρατήρησε τις λίστες παραγγελιών.

**Αναμενόμενο**

1. Οι τιμές ανταποκρίνονται στην επιλεγμένη τοποθεσία.

---

### UC-D03 — Νέο κατάστημα και όρια πλάνου

**Σκοπός:** Ομάδα growth—όρια free vs Pro.

**Βήματα**

1. Από τη λίστα τοποθεσιών δοκίμασε **Προσθήκη καταστήματος**.
2. Αν ήδη είσαι στο όριο free, δες το σφάλμα.

**Αναμενόμενο**

1. Όταν επιτρέπεται, δημιουργείται και εμφανίζεται στη λίστα.
2. Στο όριο: κατανοητό μήνυμα **upgrade σε Pro**.

# Ομάδα E — Ειδοποιήσεις τιμής (Pro)

### UC-E01 — Δημιουργία από προϊόν

**Σκοπός:** Passive monitoring value.

**Αναμενόμενο**

1. Μήνυμα επιβεβαίωσης που εξηγεί ροή λίστας / ειδοποιήσεις ανά τι έχει γραφτεί στο προϊόν.
2. Alert εμφανίζεται στη λίστα **Ειδοποιήσεις τιμής**.

---

### UC-E02 — Διαχείριση λίστας ειδοποιήσεων

**Βήματα**

1. Tab **Λογαριασμός** → πάτησε τη γραμμή **«Ειδοποιήσεις τιμής»** (ή από τη λεπτομέρεια προϊόντος με τη σύντομη συντόμευση «Λίστα ειδοποιήσεων» όταν εμφανίζεται).
2. Ενεργοποίησε/απενεργοποίησε alerts, δοκίμασε διαγραφή αν υπάρχει.

**Αναμενόμενο**

1. Η λίστα συγχρονίζεται με τον server.
2. Το copy εξηγεί ότι ο **επαναλαμβανόμενος έλεγχος** γίνεται στον **server** και ότι **push** / **email digest** εξαρτώνται από ρυθμίσεις περιβάλλοντος (δες [`TESTING_GUIDE.md`](TESTING_GUIDE.md), ενότητα «Γνωστά όρια του demo»).

**Παρουσίαση:** Μην υπόσχεσαι live push στο στάδιο LAN αν δεν έχεις ρυθμίσει APN· πες «στο παραγωγικό profile ενεργοποιείται το κανάλι.»

---

### UC-E03 — Διακόπτης email digest (Λογαριασμός)

**Σκοπός:** Να φαίνεται ότι ο χρήστης ελέγχει το **συγκεντρωτικό email** για χτυπήματα price alerts.

**Βήματα**

1. Συνδεδεμένος buyer, tab **Λογαριασμός**.
2. Βρες το block **«Email digest χτυπημάτων»**.
3. Άλλαξε το **Switch** on/off.

**Αναμενόμενο**

1. Η προτίμηση αποθηκεύεται (χωρίς crash)· μετά από refresh παραμένει η νέα τιμή.
2. **Πραγματική αποστολή email** στο demo εξαρτάται από outbound mail (π.χ. Resend) στο server — χωρίς ρύθμιση περιμένεις μόνο UI persistence, όχι γράμμα στο inbox.

## 4. Μην ξεχνάς στη live παρουσίαση

1. **Pro** στο demo **δεν** χρεώνει κάρτα — το λες ρητά.
2. **Push / email** price alerts **εξαρτώνται** από production-style config—μην τα παρουσιάζεις σαν εγγυημένα στο LAN χωρίς έλεγχο.
3. Αν κάτι αποτύχει: ακολούθησε **UC-A08** narrative — «στο deployment παραγωγής ο server είναι πάντα on».
