# Design Plan — HORECA B2B Mobile Marketplace

## Σκοπός προϊόντος

Η εφαρμογή σχεδιάζεται ως ένα **mobile-first B2B εργαλείο προμηθειών** για επαγγελματίες της εστίασης. Ο βασικός στόχος δεν είναι απλώς η αναζήτηση προϊόντων, αλλά η **γρήγορη και αξιόπιστη επανάληψη παραγγελιών**, η ανακάλυψη νέων προμηθευτών και η καθημερινή λειτουργική εξυπηρέτηση ενός επαγγελματία που κινείται διαρκώς μέσα στο κατάστημα, στην αποθήκη ή μεταξύ σημείων πώλησης. Το UI πρέπει να θυμίζει **προϊόν πρώτης κατηγορίας iOS**, με καθαρή ιεραρχία, μεγάλες επιφάνειες αφής, περιορισμένο οπτικό θόρυβο και σαφή προτεραιότητα στις ενέργειες που εκτελούνται με το ένα χέρι.

Ο αρχικός σχεδιασμός αφορά χρήση σε **portrait 9:16**, με κύριο test device το **iPhone 14 Pro Max**. Η δομή θα παραμείνει συμβατή και με Android αργότερα, αλλά το πρώτο design baseline θα ακολουθεί κυρίως τα πρότυπα του iOS, τόσο σε spacing όσο και σε πλοήγηση, feedback και οπτική γλώσσα.

## Σχεδιαστικές αρχές

Η εμπειρία πρέπει να στηρίζεται σε τέσσερις βασικές αρχές: **ταχύτητα**, **εμπιστοσύνη**, **αναγνωσιμότητα** και **επαναληψιμότητα**. Η εφαρμογή πρέπει να επιτρέπει στον χρήστη να βρει αυτό που χρειάζεται μέσα σε λίγα δευτερόλεπτα, να καταλαβαίνει αν ένας προμηθευτής είναι αξιόπιστος, να βλέπει κρίσιμες πληροφορίες χωρίς γνωστικό φορτίο και να επαναλαμβάνει παραγγελίες χωρίς περιττά βήματα.

Οι βασικές ενέργειες θα τοποθετούνται χαμηλά στην οθόνη όπου είναι εφικτό, ώστε να ευνοείται η χρήση με έναν αντίχειρα. Η πάνω περιοχή της οθόνης θα χρησιμοποιείται κυρίως για τίτλους, κατάσταση λογαριασμού, φίλτρα υψηλού επιπέδου και συνοπτική πληροφόρηση, ενώ οι ουσιαστικές ενέργειες αγοράς και παραγγελίας θα μεταφέρονται στο χαμηλότερο οπτικό πεδίο.

| Supplier Products | Βασική διαχείριση καταλόγου | προϊόντα, διαθεσιμότητα, τιμές | επεξεργασία, ενεργοποίηση/απενεργοποίηση |

## Περιεχόμενο και λειτουργικότητα ανά ρόλο

Η εφαρμογή θα έχει δύο βασικούς ρόλους: **αγοραστής** και **προμηθευτής**. Στην πρώτη έκδοση του MVP, ο αγοραστής θα είναι ο βασικός πρωταγωνιστής της εμπειρίας, επειδή εκεί εντοπίζεται το μεγαλύτερο καθημερινό operational pain. Ο προμηθευτής θα έχει επίσης mobile πρόσβαση, αλλά αρχικά με πιο συνοπτικό dashboard, έτσι ώστε να βλέπει νέες παραγγελίες, τη ροή εκτέλεσης και τη βασική κατάσταση του καταλόγου.

| Ρόλος | Κύρια ανάγκη | Σχεδιαστική προτεραιότητα | MVP scope |
| --- | --- | --- | --- |
| Buyer | γρήγορη εύρεση προμηθευτή και επανάληψη παραγγελίας | χαμηλό friction, σαφή φίλτρα, μεγάλες CTA περιοχές | πλήρες βασικό flow |
| Supplier | γρήγορη αποδοχή και διαχείριση παραγγελίας | καθαρή λίστα, status visibility, γρήγορα actions | συνοπτικό operational flow |

## Πλοήγηση εφαρμογής

Για το αρχικό MVP προτείνεται **bottom tab navigation** με τέσσερις έως πέντε κύριες περιοχές. Η επιλογή αυτή εξυπηρετεί το iPhone 14 Pro Max, επειδή τα βασικά σημεία πλοήγησης παραμένουν προσβάσιμα χαμηλά στην οθόνη. Το tab bar θα είναι σταθερό, καθαρό και χωρίς υπερφόρτωση.

| Tab | Περιγραφή |
| --- | --- |
| Home | dashboard, quick actions, προτάσεις και σύντομη εικόνα δραστηριότητας |
| Suppliers | κατάλογος προμηθευτών και φίλτρα αναζήτησης |
| Orders | ενεργές και προηγούμενες παραγγελίες |
| Favorites | αποθηκευμένοι προμηθευτές και προϊόντα για γρήγορη επανάληψη |
| Account | στοιχεία χρήστη, επιχείρηση και ρυθμίσεις |

Για τον supplier ρόλο, το Home tab θα προσαρμόζεται δυναμικά σε **Supplier Dashboard**, ενώ τα υπόλοιπα tabs θα εστιάζουν σε Orders, Products και Account. Σε μετέπειτα φάση μπορούμε να προσθέσουμε role-based tab sets, αλλά στο MVP είναι προτιμότερο να παραμείνει μία συνεπής πλοήγηση με conditional περιεχόμενο.

## Key user flows

### Buyer flow: πρώτη παραγγελία

Ο χρήστης ανοίγει την εφαρμογή, βλέπει σύντομη παρουσίαση της αξίας της, επιλέγει ότι είναι επαγγελματίας αγοραστής, δημιουργεί λογαριασμό ή συνδέεται και μεταφέρεται στο Home Dashboard. Από εκεί μπορεί να μεταβεί στον κατάλογο προμηθευτών, να εφαρμόσει φίλτρα ανά κατηγορία, να δει το προφίλ ενός προμηθευτή, να περιηγηθεί στον κατάλογο προϊόντων, να επιλέξει προϊόντα και ποσότητες, να ελέγξει το καλάθι του και να ολοκληρώσει την παραγγελία.

### Buyer flow: γρήγορη επανάληψη

Ο χρήστης ανοίγει την εφαρμογή και βλέπει στο dashboard ένα card με πρόσφατες παραγγελίες. Επιλέγει μια προηγούμενη παραγγελία, βλέπει τη σύνοψη, πατάει reorder, γίνονται μικρές προσαρμογές στις ποσότητες και συνεχίζει άμεσα στο checkout. Αυτή είναι μία από τις πιο κρίσιμες ροές για το προϊόν και θα πρέπει να είναι εμφανής από την πρώτη έκδοση.

### Supplier flow: διαχείριση νέας παραγγελίας

Ο προμηθευτής συνδέεται και μεταφέρεται σε supplier dashboard με σύνοψη νέων παραγγελιών. Ανοίγει τη λίστα incoming orders, επιλέγει μια νέα παραγγελία, ελέγχει τα προϊόντα, επιβεβαιώνει διαθεσιμότητα και αλλάζει το status σε confirmed ή preparing. Ο στόχος είναι αυτή η ροή να απαιτεί ελάχιστα taps και να είναι απολύτως σαφής ως προς το status.

## Layout κατευθύνσεις ανά βασική οθόνη

| Οθόνη | Layout οδηγία |
| --- | --- |
| Welcome | μεγάλο hero statement στο πάνω μισό, δύο ξεκάθαρα CTA στο κάτω μισό, μικρό trust strip για προμηθευτές/κατηγορίες |
| Sign In / Sign Up | card-like form layout με έντονο primary button στο χαμηλό τμήμα της οθόνης |
| Home Dashboard | πάνω μέρος με χαιρετισμό και context, μεσαίο τμήμα με quick actions, χαμηλότερα recent orders και supplier recommendations |
| Supplier Directory | sticky search bar, horizontal filter chips, vertical card list με καθαρή ιεραρχία τιμής/αξιολόγησης/κατηγοριών |
| Supplier Profile | header με identity και trust markers, μετά sections για προϊόντα, όρους συνεργασίας και CTA παραγγελίας |
| Product Catalog | compact cards για ταχύτερη σάρωση, φίλτρα κοντά στον αντίχειρα όπου είναι εφικτό |
| Cart / Checkout | καθαρό summary, εμφανές total, persistent CTA επιβεβαίωσης στο κάτω μέρος |
| Orders | segmented control για status και λίστα με έντονα badges κατάστασης |

## Χρωματική κατεύθυνση

Η χρωματική ταυτότητα πρέπει να αποπνέει **εμπιστοσύνη, επαγγελματισμό και καθαρότητα**, χωρίς να χάνει τη σύνδεσή της με τον κόσμο της εστίασης και της προμήθειας. Επιλέγω μια παλέτα που ισορροπεί ανάμεσα σε τεχνολογικό ύφος και λειτουργική σοβαρότητα.

| Χρήση | Χρώμα | Ρόλος |
| --- | --- | --- |
| Primary | `#0F766E` | βασικά call to action, active states, βασικά links |
| Secondary Accent | `#C2410C` | highlights για προσφορές, limited stock, εμπορικά σημεία προσοχής |
| Background | `#F8FAFC` | κύριο ουδέτερο υπόβαθρο με υψηλή αναγνωσιμότητα |
| Surface | `#FFFFFF` | κάρτες, forms, elevated surfaces |
| Text Primary | `#0F172A` | κύριο κείμενο |
| Text Secondary | `#475569` | υποστηρικτικό κείμενο |
| Border | `#E2E8F0` | separators, cards και input outlines |
| Success | `#15803D` | επιβεβαιώσεις και θετικά status |
| Warning | `#D97706` | pending states και λειτουργικές ειδοποιήσεις |
| Error | `#DC2626` | αποτυχίες ή κρίσιμες ενδείξεις |

Η primary απόχρωση στο πράσινο-τεαλ υποστηρίζει τη λογική της αξιοπιστίας και της καθημερινής λειτουργίας, ενώ η θερμή secondary accent φέρνει μια διακριτική εμπορική ένταση, χρήσιμη για προσφορές και urgency χωρίς να μετατρέπει το προϊόν σε καταναλωτικό e-commerce.

## Τυπογραφία και οπτική ιεραρχία

Η τυπογραφία πρέπει να είναι καθαρή, με ευανάγνωστα μεγέθη και αρκετό line-height. Οι τίτλοι θα είναι σύντομοι και λειτουργικοί. Το προϊόν δεν χρειάζεται «διαφημιστική» υπερβολή, αλλά επιχειρησιακή σαφήνεια. Οι αριθμοί, τα totals, οι ποσότητες και τα status labels πρέπει να ξεχωρίζουν οπτικά, ώστε να μπορούν να διαβαστούν γρήγορα μέσα σε συνθήκες πίεσης.

## MVP προτεραιότητες σχεδιασμού

Στην πρώτη έκδοση θα δοθεί έμφαση στις ροές που έχουν τη μεγαλύτερη επιχειρησιακή αξία. Αυτό σημαίνει ότι το MVP θα επικεντρωθεί σε welcome/authentication, buyer dashboard, supplier directory, product catalog, cart, checkout και orders. Η supplier εμπειρία θα καλύψει τις απολύτως βασικές ανάγκες αποδοχής και παρακολούθησης παραγγελιών. Πιο σύνθετες δυνατότητες, όπως πλήρες in-app chat, advanced analytics, live logistics και AI recommendations, θα έχουν προετοιμασμένη θέση στο architecture, αλλά δεν θα αποτελέσουν μέρος της πρώτης παράδοσης.

## Σημειώσεις για επόμενη φάση

Η επόμενη πρακτική φάση μετά το design είναι να μετατραπεί αυτή η εμπειρία σε συγκεκριμένο **backlog υλοποίησης**, ώστε να υλοποιηθούν σταδιακά το navigation shell, τα mock datasets, τα role-based screens, το branding και τα πρώτα reusable components. Παράλληλα θα χρειαστεί να οριστεί από νωρίς η ορολογία του domain, ώστε έννοιες όπως supplier, MOQ, reorder, delivery window και order status να αποτυπώνονται με συνέπεια σε όλη την εφαρμογή.
